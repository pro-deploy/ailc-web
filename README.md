# ailc-web

Продающий одностраничный лендинг ailc и каналы доставки MCP-сервера «без сборки». Этот
репозиторий не содержит исходного кода ядра ailc: ядро живёт рядом, в репозитории
[`pro-deploy/ailc`](https://github.com/pro-deploy/ailc), и собирается там. Здесь же
находится витрина продукта и упаковка, которая позволяет любому пользователю подключить
ailc в редактор одной строкой, не компилируя ничего локально.

## Зачем это нужно

У ailc есть три способа использования, и все три ведут к одному результату, то есть к
детерминированному гейту качества и безопасности прямо в редакторе кода.

Первый способ, самый простой, это вписать несколько строк в конфигурацию редактора. При
первом обращении к серверу готовый бинарь нужной платформы скачивается сам из релиза на
GitHub, сервер инициализирует проект, и следующий запрос к нейросети уже проходит через
гейт. Ни сборки, ни возни со скриптами.

Второй способ, для тех, у кого нет Node.js, это контейнер Docker с уже вложенным бинарём.
Папка проекта монтируется в контейнер, и всё работает локально.

Третий способ, для тех, кто хочет полного контроля, это форк репозитория ядра и сборка из
исходников. Этот путь описан в самом репозитории ailc.

Лендинг продаёт продукт и выдаёт готовые к копированию конфигурации под все эти каналы.

## Важная техническая граница

MCP-сервер ailc анализирует локальные файлы проекта по переданному пути. Чтобы видеть ваш
код, сервер обязан исполняться на вашей машине. Поэтому «онлайн» здесь означает не сервер
в облаке, а готовый бинарь, который не нужно компилировать: вы добавляете строку в
конфигурацию, и бинарь скачивается сам. Облачный сервер не видит локальные файлы, так как
протокол MCP файловую систему не передаёт. Лендинг при этом обычная статика и разворачивается
где угодно.

## Состав репозитория

Каталог `site` содержит сам лендинг: `index.html`, стили `styles.css` и небольшой скрипт
`app.js` без внешних зависимостей (переключение редактора и способа доставки, подсветка
конфигурации, кнопка копирования, проявление секций при прокрутке).

Файл `Dockerfile` в корне собирает образ MCP-сервера. Образ не компилирует Rust: он
скачивает готовый бинарь нужной архитектуры из релиза, как это делают обёртка npx и
установочные скрипты. Образ предназначен для локального запуска рядом с проектом.

Файл `docker-compose.yml` в корне публикует лендинг одной командой: собирает образ из
`deploy/Dockerfile.web` (nginx со статикой) и поднимает сайт на 80 порту. Каталог `deploy`
содержит детали этой сборки: `Dockerfile.web` и конфигурацию `nginx.conf`.

Каталог `connect` содержит готовые конфигурации подключения под Claude Code, Cursor, VS Code
с агентом GitHub Copilot, Cline, а также вариант через Docker.

Каталог `.github/workflows` содержит конвейер `docker.yml`, который собирает многоархитектурный
образ MCP-сервера и публикует его в GitHub Container Registry.

## Запуск лендинга локально

Самый быстрый способ посмотреть сайт это открыть `site/index.html` в браузере. Ближе к боевой
среде поднимите статику через Docker на удобном порту:

```sh
PORT=8080 docker compose up -d --build
# сайт на http://localhost:8080
```

Либо любым статическим сервером, например `python3 -m http.server` из каталога `site`.

## Публикация на сервере

Лендинг это статические файлы, поэтому публикация сводится к одной команде. На сервере с
установленным Docker:

```sh
git clone https://github.com/pro-deploy/ailc-web && cd ailc-web
docker compose up -d --build
```

Сайт поднимется на 80 порту сервера. Контейнер сам перезапускается после перезагрузки сервера
(`restart: unless-stopped`) и проверяется на живость (healthcheck). Поменять порт можно
переменной `PORT`, например `PORT=8080 docker compose up -d --build`. Обновить сайт после
изменений: `git pull && docker compose up -d --build`.

Для домена и автоматического протокола HTTPS поставьте перед контейнером обратный прокси
(Caddy, Traefik или nginx с certbot). Как альтернатива Docker, статику из каталога `site`
можно положить на любой статический хостинг: GitHub Pages, Cloudflare Pages, Netlify, Vercel.

## Публикация образа MCP-сервера

Образ собирается и публикуется конвейером `.github/workflows/docker.yml` при публикации тега
вида `vX.Y.Z` или вручную через «workflow_dispatch» с указанием версии. Образ выкладывается в
`ghcr.io/<владелец>/ailc`. Версия образа должна соответствовать опубликованному релизу бинаря
ailc, потому что образ скачивает бинарь именно этой версии.

Собрать образ вручную:

```sh
docker build -t ghcr.io/pro-deploy/ailc:0.4.1 .
docker run -i --rm -v "$PWD:/work" -w /work ghcr.io/pro-deploy/ailc:0.4.1 dod .
```

## Подключение в редактор

Скопируйте конфигурацию из каталога `connect` или из раздела «Подключение» на лендинге.
Например, для Claude Code в файл `.mcp.json` в корне проекта:

```json
{ "mcpServers": { "ailc": { "command": "npx", "args": ["-y", "ailc-mcp"] } } }
```

Для варианта без Node.js используйте конфигурацию через Docker из `connect/docker.mcp.json`.

## Лицензия

Открытое ядро ailc распространяется под лицензией Apache 2.0. Этот репозиторий с лендингом и
упаковкой следует той же лицензии.
